# Product Requirement Document
(PRD)

## 1. 프로젝트 개요 (Project Overview)

- **프로젝트 명**: Orbit (자미두수로 자신을 이해하다)
- **목표**: 자미두수 명리학 기반의 초개인화 AI 분석을 통해, 유저의 고유한 성향과 매력, 장단점을 진단하고 시기별 맞춤형 라이프스타일 가이드라인을 제시.
- **핵심 가치 1 (차별화된 경험 및 분석)**: 몽환적이고 직관적인 UI/UX와 고도화된 AI 알고리즘을 결합하여, 높은 정확도의 인사이트와 몰입감 있는 서비스 경험 제공.
- **핵심 가치 2 (개인화된 커리어 방향성 제시)**: 유저의 잠재력과 강점을 극대화할 수 있는 최적의 진로 및 직업 추천.
- **핵심 가치 3 (매력 자산 기반 관계 가이드)**: 개인의 고유한 매력 요소를 분석하고 이를 활용한 맞춤형 연애 및 대인관계 조언 제공.
- **핵심 가치 4 (맞춤형 여가 및 웰니스 제안)**: 유저의 성향을 바탕으로 심리적 만족도와 행복감을 높일 수 있는 취미 활동 추천.
- **핵심 가치 5 (시기별 전략적 라이프 플래닝)**: 운세 흐름에 맞춰 특정 시기에 대비하고 실행해야 할 핵심 액션 플랜 제공.

## 2. 타겟 유저(Target Audience)

- **Core Target (핵심 타겟)**: 운세 및 사주 등 명리학 기반의 자기 탐구와 자아실현에 관심도가 높은 2030 MZ세대 및 자미두수를 처음 접하지만 합리적인 비용으로 가볍게 경험해보고 싶은 입문자.
- **Behavioral Target (행동적 타겟)**: 모바일 환경에 친숙하며, 복잡한 텍스트 위주의 해설보다는 직관적인 UI/UX와 시각화된 데이터를 통해 초개인화된 분석 결과를 소비하고자 하는 사용자.

## 3. 기술 스택(Tech Stack)

- **Web Framework**: Next.js 16.2.6(App Router)
- **Language**: TypeScript
- **Styling**: Tailwind CSS, Shadcn/ui
- **Backend & DB**: Next.js API Routes, Supabase
- **Payments**: Toss Payments
- **AI**: Gemini with gemini sdk
- **자미두수 명반 검색 엔진**: https://github.com/rath/orrery.git

## 4. 디자인 가이드(Design Guide)

- **컨셉**: "Cosmic Constellation" (칠흑 우주와 빛나는 별자리)
- **배경**: Deep Space Black (#050510)과 Nebula Orange/Indigo 그라데이션
- **요소**: 우주 유리판(Glassmorphism), 반짝이는 별 배경 애니메이션, 유성 효과
- **포인트 컬러**: Cosmic Orange (#FF6B35), Warm Amber (#FFAB40), Gold Glow (#FFD700)
- **폰트**: 가독성 좋은 Sans-serif 계열 (Pretendard) + 서비스명 포인트 폰트

### 6.2 유저 UX 페이지 
1. 랜딩 페이지(/)

- 성별 선택 (남성/여성)

- CTA 버튼 (명반 생성): "내 별빛 이야기 들어보기" - 클릭 시 명반 페이지로 이동

2. 자미두수 명반 페이지 (/result)
- **[네비게이션]** 왼쪽 상단 뒤로가기 버튼 배치 (이전 상태 보존)
- 생년월일기반 자미두수 명반 렌더링 (블러 없이 전체 공개)
  - **[명반 시각화 고도화]** 주성(14정성)은 수직 배열(Amber 색상)로 가독성을 높이고, 보조성(길성/흉성)은 각각 다른 색상(Emerald/Rose)으로 배치합니다. 사화(화록/권/과/기) 속성이 있는 별은 우측에 뱃지(`[화록]` 등)를 달아 시각적 구분을 명확히 합니다. 특히 모바일 기기 등 작은 화면에서 각 궁의 텍스트가 겹치지 않도록 가변 폰트(`text-[9px]` 등) 적용 및 스크롤 영역을 분리하는 반응형 UI 개선을 완료했습니다.
- **[무료 기초 해석 - DB 매핑 데이터 활용]**
  - 유저의 명궁(Life Palace) 주성을 파악하여 진짜 모습을 렌더링. (예: "당신의 진짜 모습을 결정짓는 핵심 별자리입니다")
  - 명궁에 주성이 없는 경우(명궁무주성), 천이궁의 주성 및 보조성/사화를 차용하며 "매우 특별하고 흥미로운 진짜 모습"임을 강조.
- **[운세 시기 탐색 타임라인 & 가이드 모달]**
  - 명반 하단에 100세까지의 대운(대한)과 해당 대운에 속하는 연도(유년)를 선택할 수 있는 2단 가로 스크롤 타임라인 칩 UI 제공.
  - 연도(유년) 선택 시, 브라우저 단에서 실시간으로 계산을 수행해 명반 그리드 상의 해당 궁위에 `[10년운]`(주황색), `[1년운]`(초록색) 배지를 띄우고 네온 하이라이트 효과를 연동.
  - "운세 보는 법" 모달 창 제공: 12궁, 14주성, 보조성, 사화(화록/권/과/기)의 실질적인 운세적 의미와 길흉의 증폭 작용을 상세히 설명하는 용어 사전 탑재.
- 테마 선택 (결제 옵션): 나의 잠재력과 커리어(990원), 나만의 매력과 관계(990원), 나를 채우는 여가와 웰니스(500원)
  - **[UI 개선 완료]** 테마별 서브 텍스트를 '나 활용법'에 맞추어 렌더링 (예: 커리어 활용법, 관계 활용법, 라이프 활용법)
  - **[테마별 분석 궁 매핑]**
    - 커리어: 명궁 + 관록궁(업무 스타일) + 재백궁(재물관) + 화록/록존 위치 궁(숨겨진 금광=수익 파이프라인)
    - 연애/매력: 명궁 + 부처궁(관계 태도) + 천이궁(대인관계) + 자녀궁(본능적 매력/도화)
    - 여가/웰니스: 명궁 + 질액궁(육체적 건강/에너지) + 복덕궁(정신적 만족도/취미)
- 결제 폼 이동 CTA 버튼: "선택한 테마의 내 별빛 이야기 들어보기"
  - **[반응형 폰트]** 기기 화면 너비에 맞게 버튼 텍스트와 아이콘 크기가 자동으로 조절되도록 구현 완료

3. 주문 작성 페이지 (/order-form)
- **[네비게이션]** 왼쪽 상단 뒤로가기 버튼 배치 (이전 상태 보존)
- 타이틀 및 안내: "내 별빛 이야기 기록하기 - 언제든 다시 꺼내볼 수 있도록 연락처와 비밀번호를 남겨주세요."
- **[내 별빛 이야기 정보]** 유저의 생년월일, 시간, 성별, 지역 및 선택한 테마/금액 요약 정보 렌더링
- **[유저 정보 입력]** 전화번호(자동 하이픈 포맷팅) 및 비밀번호(향후 별빛 이야기 조회용) 입력란
  - *비밀번호 갱신 편의성:* 기존 가입된 전화번호로 접근 시 비밀번호가 달라도 '가입 실패' 에러를 띄우지 않고, 즉시 새로운 비밀번호로 계정 정보를 덮어쓰기(갱신)한 뒤 로그인되도록 로직 개선 (`generateLink` Admin API 활용 유저 조회).
- 결제 이동 CTA 버튼: "내 별빛 이야기 보러가기" (클릭 시 결제 페이지(`/checkout`)로 이동하며 데이터 보존)

4. 결제 페이지 (/checkout)
- **[뒤로가기]** 주문서로 돌아가기 버튼 배치
- 타이틀 및 안내: "내 별빛 이야기 결제 - 별빛 이야기 해독을 위한 마지막 단계입니다."
- **[주문 상세 정보]** 유저의 연락처 및 선택한 상품(테마) 정보 요약 렌더링
- **[금액 동적 연동]** 테마 옵션에 따라 결제 금액 동적 계산 (커리어/연애: 990원, 취미: 500원)
- **[토스페이먼츠 연동 (v2)]** 최신 SDK 기반의 결제수단(PaymentWidget) 및 이용약관 렌더링 (결제 버튼은 명확한 '금액 결제하기' 텍스트 사용 권장)
  - *UI 개선:* CSS 필터를 적용하여 프로젝트 우주 테마와 어울리는 커스텀 다크 모드 구현
  - *동기화 최적화:* 주문 정보를 백엔드에서 비동기로 완벽히 로드한 직후에 위젯을 초기화하도록 의존성 배열(Dependency Array) 렌더링 로직 개선 (무한 로딩 버그 픽스)
- 결제 프로세스 콜백 리다이렉트 처리:
  - 승인 완료 시 `/checkout/success` (별빛 이야기 해독 준비 완료 안내 및 '내 별빛 이야기 보러가기' 버튼 렌더링)
    - *UI 개선:* 연락처 누락 버그 해결(URL 쿼리로 상태 전달) 및 모바일 환경 주문번호 강제 개행(줄바꿈) 방지 레이아웃 적용 완료.
  - 결제 실패/취소 시 `/checkout/fail` (오류 사유 안내 및 '다시 별빛 이야기 시작하기' 버튼 렌더링)
    - *예외 처리:* 에러가 "이미 처리된 결제(`ALREADY_PROCESSED_PAYMENT`)"인 경우, 에러 화면을 노출하거나 알림을 발송하지 않고 메인 랜딩 페이지(`/`)로 자동 리다이렉트 처리.

## 7. API 구조 (API Architecture)

본 프로젝트는 Next.js 14 App Router의 **Server Actions**를 주력으로 사용하여 내부 비즈니스 로직을 처리하며, 외부 연동(결제 웹훅 등)에만 API Routes를 혼용하여 구성합니다.

### 7.1 Auth API (인증/인가)
비회원 유저와 관리자의 인증 상태를 관리합니다. (Server Actions로 구현됨)
- `loginAction` : 비회원 로그인 (전화번호, 비밀번호 기반 검증, 5회 실패 시 5분 잠금 및 Supabase 세션 자동 발급)
  - *개선사항:* Client Component의 `try-catch` 블록 내에서 Server Action의 `redirect()` 호출 시 발생하는 `NEXT_REDIRECT` 에러를 방지하기 위해 성공 여부(`success: true`)만 반환하고 라우팅은 클라이언트 위임 처리.
- `logoutAction` : 로그아웃 및 세션 파기
- `adminLoginAction` : 관리자 로그인 (관리자 권한 검증 및 세션 발급)

### 7.2 Myeongban API (명반 조회 및 기초 해석 데이터 제공)
결제 전 명반 페이지 렌더링용 자미두수 데이터를 생성하고, 무료 해석 섹션을 위한 기초 데이터를 제공합니다. (Server Action으로 구현됨)
- `getMyeongbanAction` : 생년월일시 데이터를 받아 `@orrery/core`로 명반 데이터 계산. 명궁(Life Palace)의 주성을 파악(명궁무주성 시 천이궁 주성 차용)하여 `z_knowledge_base` 테이블에서 해당 주성의 `core_trait`을 조회 후 프론트에 전달 (보조성 묶음 데이터와 혼동되지 않도록 정확한 단일 주성 매칭 로직 적용).

### 7.3 Orders API (주문 관리 및 Toss Payments 연동)
주문서 생성, 결제 정보 검증 및 Toss Payments 승인 연동을 Server Action으로 처리합니다.
- `createOrderAction`: 전화번호와 비밀번호를 받아 가상 이메일(`u{phone}@orbit-app.com`) 및 패딩된 비밀번호(`{password}_orbit`)로 백그라운드 회원가입/로그인(이메일 인증 우회)을 처리한 뒤, `public.orders`에 `pending` 상태의 주문을 생성합니다. 주문 생성 시 `@orrery/core`를 활용해 명반을 선계산하여 주성(14정성), 보조성(길성/흉성 14개), 사화(화록/화권/화과/화기)를 추출 및 차성안궁 로직을 거쳐 `saju_data`에 함께 보관합니다.
- `getOrderAction`: `orderId`를 기반으로 결제 금액과 테마 등 주문서 상세 정보를 조회하여 클라이언트(결제창)에 전달합니다.
- `confirmPaymentAction`: 토스페이먼츠 결제창(위젯) 인증 통과 후 리다이렉트되는 성공 페이지에서 호출되며, 서버 대 서버로 Toss Confirm API를 호출하여 결제를 승인하고 `orders`와 `payments` 테이블을 업데이트합니다. (Toss 리다이렉트에 의한 쿼리 파라미터 중복/배열화 문제 방어 처리 포함)
  - *보안/안정성:* 3rd Party 결제 모듈(Toss) 리다이렉트 시 브라우저 정책(SameSite)으로 인해 발생하는 세션 유실 및 RLS(Row Level Security) 접근 제한 에러 방지를 위해, 결제 검증 후 DB 업데이트 시 Admin Client(`SERVICE_ROLE_KEY`)를 사용하여 강제 업데이트 처리.
  - **[텔레그램 알림 연동]**: 결제 성공 또는 실패 시 백그라운드에서 지정된 텔레그램 채팅방으로 즉시 알림 메시지 발송 처리 추가. Vercel 서버리스 환경에서 로직이 일시 정지되는 것을 막고, 정확한 타이밍에 발송되도록 반드시 `await` 처리하여 동기화.

### 7.4 Payments API (결제 연동)
토스페이먼츠(v2 SDK)와의 결제 승인 및 검증 로직을 처리합니다.
- `POST /api/payments/confirm` : 결제 승인 요청 (토스페이먼츠 서버로 `paymentKey`, `orderId`, `amount` 전송 및 검증). 결제 성공 시 주문 상태(status) 업데이트 및 `Reports API` 호출 트리거
- `POST /api/payments/webhook` : 결제 취소, 가상계좌 입금 등 비동기 웹훅 이벤트 처리

### 7.5 Reports API (자미두수 리포트)
Gemini AI 기반의 분석 결과 리포트 생성을 담당합니다.
- `GET /api/reports/[id]` : 생성된 자미두수 분석 리포트 내용 조회
- `GET /api/reports/shared/[id]` : 외부 공유된 자미두수 리포트 퍼블릭 조회
- `POST /api/reports/[id]/generate` : 주문 정보를 바탕으로 Gemini API를 호출하여 자미두수 분석 결과 (재)생성 (결제 완료 시 자동 호출). 
  - **프롬프트 전략 고도화 (Jargon-Free & Periodic Theme)**: 유저에게 명리학 전문 용어(명궁, 화기, 보조성 등) 노출을 100% 차단하고 MBTI/심리테스트 스타일의 친숙한 일상어로 치환합니다. 또한, 선택한 테마(예: 커리어)에 해당하는 **대한(10년 운) 및 유년(1년 운) 테마 궁**의 별 정보도 추출해 RAG 컨텍스트에 주입하여, 시기적절하고 디테일한 미래 예측을 제공합니다.
  - **테마별 맞춤형 시스템 프롬프트 분리**: 공통 프롬프트(전문 용어 금지, JSON 스키마 준수)를 기반으로 테마별 특화 프롬프트를 분리 적용합니다.
    - 커리어 프롬프트: 커리어 전략가 역할. 관록궁/재백궁 기반 업무 스타일 분석 + 화록/록존이 위치한 궁을 동적 탐색하여 "숨겨진 금광(수익 파이프라인)"으로 구체적 비즈니스 모델(대중/팬덤 기반, 부동산/공간 기반, 대기업/기관 기반, 콘텐츠/교육 기반 등) 제시.
    - 연애/매력 프롬프트: 관계 심리 분석가 & 매력 컨설턴트 역할. 부처궁 기반 관계 태도 분석 + 자녀궁 데이터를 "본능적 매력 자산"으로 라벨링하여 도화 매력, 플러팅 스타일, 성적 어필 포인트 분석.
    - 여가/웰니스 프롬프트: 라이프스타일 큐레이터 & 멘탈케어 전문가 역할. 질액궁 기반 육체적 에너지 관리 + 복덕궁 기반 맞춤형 취미/여가 큐레이션.
  - **자가복구(Self-Healing) 로직 도입**: 결제 완료 후 일시적인 Gemini API 통신 오류(타임아웃 등) 발생 시 리포트 생성이 누락되는 것을 방지하기 위해 최대 5회 자동 재시도(Retry) 및 백오프(1.5초 대기) 로직 구현. 성공 시 즉각 중복 요청 방지.
  - **[Vercel 무한 로딩 해결]**: Vercel의 Serverless Function 생명주기 제약으로 인해 결제 완료 시점(`payment.ts`)이 아닌, 유저가 리포트 페이지(`/reports/[id]`)에 진입하여 대기 화면을 볼 때 **클라이언트 측에서 명시적으로 리포트 생성 서버 액션을 호출**하도록 구조 개선.
  - **[접근 권한 디버깅]**: Vercel Serverless 한계 회피를 위해 클라이언트 측에서 대기열을 생성하는 과정에서, DB `reports` 테이블의 `INSERT` / `UPDATE` RLS 정책 부재로 인한 404 차단 버그 발생. `reports_insert_owner` 및 `reports_update_owner` 정책을 추가하고 404 분기문을 수정하여 해결.
  - **[UI 렌더링 에러 차단]**: AI가 지시사항 문구(예: a, b)를 오해하여 단순 문자열(String) 대신 중첩된 객체(Object)를 반환할 때 발생하는 치명적인 프론트엔드 크래시(Minified React Error #31) 현상을 방지하기 위해, 시스템 프롬프트 제약조건 강화 및 서버 단에서 객체를 강제 문자열로 변환하는 이중 Fallback 안전장치 추가.
  - **[텔레그램 알림 연동]**: 리포트 해독 성공/실패 시 지정된 텔레그램 채팅방으로 상태 알림 자동 발송. 결제 알림과의 데이터 추적 무결성을 위해 `reportId` 대신 결제 시 사용된 `orderId`를 주입하여 알림 내용 통일 완료.

### 7.6 Admin API (관리자 전용)
관리자 대시보드 및 전체 서비스 데이터 관리를 수행합니다.
- `GET /api/admin/dashboard` : 요약 카드 데이터(총 매출, 신규 주문, 총 유저수, 총 해석 건수) 및 기간별 매출 대시보드 데이터 조회
- `GET /api/admin/orders` : 전체 유저의 주문 내역 리스트 조회 (페이지네이션 및 상태 필터링 지원)
- `GET /api/admin/users` : 서비스를 사용하는 전체 유저 리스트 조회
- `PATCH /api/admin/orders/[id]/status` : 주문 상태 강제 변경 (취소, 환불 등)

## 8. 데이터베이스 스키마 (Database Schema)

Supabase PostgreSQL 기반으로 작성된 주요 테이블 구조입니다. (모든 테이블에는 `created_at` 컬럼이 기본적으로 포함됩니다.)

### 8.1 `users` (유저 정보)
Supabase Auth(`auth.users`)와 연결되는 사용자 정보 확장 테이블입니다.
- `id` (uuid) : PK, `auth.users.id` 참조, **NOT NULL**
- `phone_number` (varchar) : 로그인 및 식별용 전화번호, **NOT NULL**
- `role` (varchar) : 권한 ('user', 'admin'), **NOT NULL** (Default: 'user')
- `created_at` (timestamptz) : 생성일시, **NOT NULL** (Default: now())
- **[PostgreSQL Trigger]**: `auth.users`에 신규 가입자가 생기면 `handle_new_user` 트리거가 동작하여 자동으로 이 테이블에 데이터를 동기화합니다.

### 8.2 `orders` (주문서)
사주 입력 데이터, 선택 테마 및 주문의 진행 상태를 저장합니다.
- `id` (uuid) : PK, **NOT NULL**
- `user_id` (uuid) : FK (`users.id`), **NOT NULL**
- `saju_data` (jsonb) : 생년월일, 태어난 시간, 성별, 지역, 1차 추출된 명반 별자리 데이터 등, **NOT NULL**
- `theme` (varchar) : 선택 테마 ('career', 'love', 'hobby'), **NOT NULL**
- `amount` (integer) : 결제 요청 금액 (진로/연애: 990원, 취미: 500원), **NOT NULL**
- `status` (varchar) : 주문 상태 ('pending', 'paid', 'cancelled', 'refunded'), **NOT NULL** (Default: 'pending')
- `created_at` (timestamptz) : 생성일시, **NOT NULL** (Default: now())
- `updated_at` (timestamptz) : 수정일시, **NOT NULL** (Default: now())

### 8.3 `payments` (결제 정보)
토스페이먼츠(v2) 연동 시 발생하는 트랜잭션 데이터를 관리합니다.
- `id` (uuid) : PK, **NOT NULL**
- `order_id` (uuid) : FK (`orders.id`), **NOT NULL**
- `payment_key` (varchar) : 토스페이먼츠 결제 고유 키, **NULL** (승인 완료 전)
- `method` (varchar) : 결제 수단 ('카드', '가상계좌' 등), **NULL**
- `amount` (integer) : 실제 승인 금액, **NOT NULL**
- `status` (varchar) : 결제 상태 ('ready', 'done', 'canceled', 'aborted'), **NOT NULL** (Default: 'ready')
- `receipt_url` (varchar) : 영수증 URL 링크, **NULL**
- `paid_at` (timestamptz) : 결제 승인 완료 일시, **NULL**
- `created_at` (timestamptz) : 생성일시, **NOT NULL** (Default: now())

### 8.4 `reports` (분석 리포트)
Gemini AI를 통해 생성된 자미두수 분석 결과를 저장합니다.
- `id` (uuid) : PK, **NOT NULL**
- `order_id` (uuid) : FK (`orders.id`), UNIQUE, **NOT NULL**
- `content` (jsonb) : AI 분석 결과 구조화 데이터, **NULL** (생성 전)
- `status` (varchar) : 리포트 상태 ('pending', 'generating', 'completed', 'failed'), **NOT NULL** (Default: 'pending')
- `is_public` (boolean) : 외부 공유 허용 여부, **NOT NULL** (Default: false)
- `generated_at` (timestamptz) : 생성 완료 일시, **NULL**
- `created_at` (timestamptz) : 레코드 생성 일시, **NOT NULL** (Default: now())

## 9. 자미두수 역학 지식베이스 스키마 (Astrological Knowledge Base Schema)

기존의 파편화된 다중 테이블 방식을 탈피하고, AI가 추출한 5대 핵심 가치 기반의 정규화된 데이터를 담는 단일 통합 테이블 구조(Single Table Inheritance)를 사용합니다. 이를 통해 유저의 명반 데이터와 유연하게 매칭하여 AI 리포트 생성 시 최적의 프롬프트 컨텍스트를 제공합니다.

### 9.1 `z_knowledge_base` (통합 지식베이스 테이블)
자미두수의 모든 지식(주성, 보좌성, 12궁, 격국, 사화 등)을 하나의 구조로 통합 저장합니다.
- `id` (uuid) : PK, **NOT NULL**
- `category` (varchar) : 데이터 범주 ('star', 'palace', 'formation'), **NOT NULL**
- `target_subject` (varchar) : 분석 대상 제목 (예: '자미두수 거문성 명궁 및 격국'), **NOT NULL**
- `teaser_quote` (varchar) : [결제 전 후킹용] 명반 페이지에서 유저의 호기심을 유발할 수 있는 짧은 미끼 문구 (예: "타고난 리더십! 하지만 2026년에 예상치 못한 변수가...")
- `core_trait` (text) : 고유한 성향, 강점 및 잠재적 단점 요약 (핵심가치 1)
- `career_insight` (text) : 잠재력을 극대화할 수 있는 진로 및 직업 방향성 (핵심가치 2)
- `love_insight` (text) : 매력 자산 기반의 연애 및 대인관계 조언 (핵심가치 3)
- `wellness_insight` (text) : 성향에 맞는 취미, 여가, 심리적 만족도 증진 플랜 (핵심가치 4)
- `periodic_insight` (text) : 특정 시기에 대비해야 할 핵심 액션 플랜 (핵심가치 5)
- `search_vector` (tsvector) : Full-text 검색(명반과 매칭)을 위한 벡터 인덱스
- `created_at` (timestamptz) : 생성일시, **NOT NULL** (Default: now())
